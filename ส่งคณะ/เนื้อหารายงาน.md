# DIY Autonomous Mobile Robot

## บทที่ 1: บทนำ

### 1.1 ที่มาและความสำคัญ

ในปัจจุบัน หุ่นยนต์เคลื่อนที่อัตโนมัติ (Autonomous Mobile Robot: AMR) มีบทบาทสำคัญในหลากหลายอุตสาหกรรม ไม่ว่าจะเป็นงานขนส่งในคลังสินค้า การบริการในโรงพยาบาล หรือการสำรวจพื้นที่ที่เป็นอันตราย หุ่นยนต์ AMR แตกต่างจาก AGV (Automated Guided Vehicle) ตรงที่สามารถสร้างแผนที่สภาพแวดล้อม วางแผนเส้นทาง และหลบหลีกสิ่งกีดขวางได้อัตโนมัติ โดยไม่ต้องพึ่งพาเส้นทางที่กำหนดไว้ล่วงหน้า เช่น เส้นแถบแม่เหล็กหรือเทปนำทาง

อย่างไรก็ตาม หุ่นยนต์ AMR ที่มีจำหน่ายในท้องตลาดมักมีราคาสูง และมีข้อจำกัดด้านการปรับแต่ง โปรเจคนี้จึงมีวัตถุประสงค์ในการออกแบบและสร้าง AMR จากศูนย์ (from scratch) โดยใช้อุปกรณ์ที่หาได้ง่ายและมีต้นทุนต่ำ ร่วมกับซอฟต์แวร์โอเพนซอร์ส เพื่อศึกษาและทำความเข้าใจหลักการทำงานของระบบหุ่นยนต์อัตโนมัติอย่างครบถ้วน ตั้งแต่การควบคุมมอเตอร์ระดับต่ำ ไปจนถึงการสร้างแผนที่และนำทางอัตโนมัติ

### 1.2 วัตถุประสงค์

1. ออกแบบและสร้างหุ่นยนต์เคลื่อนที่แบบ Differential Drive ที่สามารถเคลื่อนที่ได้อย่างแม่นยำ
2. พัฒนาระบบ SLAM (Simultaneous Localization and Mapping) สำหรับสร้างแผนที่สภาพแวดล้อมแบบ 2 มิติ โดยใช้ LiDAR
3. พัฒนาระบบนำทางอัตโนมัติ (Autonomous Navigation) ที่สามารถวางแผนเส้นทางและหลบหลีกสิ่งกีดขวางได้
4. ศึกษาและประยุกต์ใช้ ROS2 (Robot Operating System 2) ในการพัฒนาซอฟต์แวร์หุ่นยนต์

### 1.3 ขอบเขตของโปรเจค

- หุ่นยนต์ขนาดเล็ก (16×16 ซม.) สำหรับใช้งานภายในอาคาร
- ใช้ระบบขับเคลื่อนแบบ Differential Drive (2 ล้อขับ + 1 ล้ออิสระ)
- ใช้ LiDAR 2D ในการสแกนสภาพแวดล้อม
- ทำงานบนระบบปฏิบัติการ Ubuntu 24.04 กับ ROS2 Jazzy
- สื่อสารระหว่าง Raspberry Pi 5 กับ ESP32 ผ่าน Serial (USB)

## บทที่ 2: ทฤษฎีที่เกี่ยวข้อง

### 2.1 Differential Drive Kinematics

หุ่นยนต์แบบ Differential Drive ขับเคลื่อนด้วยล้อ 2 ล้อที่หมุนอิสระต่อกัน การควบคุมความเร็วและทิศทางทำได้โดยการปรับความเร็วรอบของล้อแต่ละข้าง

**Forward Kinematics** — แปลงความเร็วล้อเป็นความเร็วหุ่นยนต์:
```
v = (v_left + v_right) / 2          ← ความเร็วเชิงเส้น (m/s)
ω = (v_right - v_left) / L          ← ความเร็วเชิงมุม (rad/s)
```
โดย L = ระยะห่างระหว่างล้อ (wheelbase)

**Inverse Kinematics** — แปลงคำสั่งความเร็วเป็นความเร็วล้อ:
```
v_left  = v - (ω × L / 2)
v_right = v + (ω × L / 2)
```

### 2.2 Odometry (การวัดระยะทางจาก Encoder)

Dead-Reckoning Odometry คำนวณตำแหน่งหุ่นยนต์จากการนับ tick ของ Encoder:
```
d_left   = Δticks_left  × distance_per_count
d_right  = Δticks_right × distance_per_count
d_center = (d_left + d_right) / 2
Δθ       = (d_right - d_left) / L

x ← x + d_center × cos(θ)
y ← y + d_center × sin(θ)
θ ← θ + Δθ
```

### 2.3 PID Controller

PID (Proportional-Integral-Derivative) Controller ใช้ในการควบคุมความเร็วล้อให้ตรงตามค่าที่สั่ง:
```
output = Kp × error + Ki × ∫error dt + Kd × (d(error)/dt)
```
- **P (Proportional):** ตอบสนองต่อ error ปัจจุบัน — ยิ่ง error มาก ยิ่งแก้มาก
- **I (Integral):** สะสม error เพื่อกำจัด steady-state error
- **D (Derivative):** ทำนาย error ในอนาคตเพื่อลด overshoot

### 2.4 SLAM (Simultaneous Localization and Mapping)

SLAM เป็นกระบวนการที่หุ่นยนต์สร้างแผนที่สภาพแวดล้อมพร้อมกับระบุตำแหน่งตัวเองในแผนที่นั้น โดยใช้ข้อมูลจาก LiDAR ในการแมตช์รูปแบบ scan กับแผนที่ที่สร้างขึ้น

ในโปรเจคนี้ใช้ **slam_toolbox** ซึ่งเป็นแพ็กเกจ SLAM ที่ออกแบบมาสำหรับ ROS2 โดยเฉพาะ ใช้อัลกอริทึม **Online Asynchronous SLAM** ที่เหมาะกับ hardware ที่มี computing power จำกัดอย่าง Raspberry Pi 5

### 2.5 Autonomous Navigation (Nav2)

Nav2 (Navigation 2) เป็น framework สำหรับการนำทางอัตโนมัติใน ROS2 ประกอบด้วย:
- **Path Planner** — วางแผนเส้นทางจากจุดเริ่มต้นถึงเป้าหมาย
- **Path Controller** — ควบคุมหุ่นยนต์ให้วิ่งตามเส้นทาง
- **Costmap** — แผนที่ต้นทุน (cost) ที่แสดงพื้นที่ว่างและสิ่งกีดขวาง
- **Behavior Tree** — ตรรกะการตัดสินใจว่าต้องทำอะไรเมื่อเกิดสถานการณ์ต่างๆ
- **Recovery Behaviors** — พฤติกรรมกู้คืนเมื่อหุ่นยนต์ติดสิ่งกีดขวาง

### 2.6 ROS2 (Robot Operating System 2)

ROS2 เป็น middleware สำหรับพัฒนาซอฟต์แวร์หุ่นยนต์ ทำหน้าที่เป็นตัวกลางในการสื่อสารระหว่าง node (โปรแกรมย่อย) ต่างๆ ผ่านระบบ:
- **Topics** — การสื่อสารแบบ publish/subscribe (เช่น `/scan`, `/odom`, `/cmd_vel`)
- **Services** — การเรียกฟังก์ชันแบบ request/response
- **Actions** — การสั่งงานระยะยาวพร้อม feedback
- **TF (Transform)** — ระบบจัดการ coordinate frame ต่างๆ ของหุ่นยนต์

### 3.3 พัฒนาซอฟต์แวร์ ESP32 — Motor Controller Firmware

Firmware บน ESP32 เขียนด้วย C++ ผ่าน PlatformIO โดยมีหน้าที่หลัก:

#### 3.3.1 การอ่าน Encoder (Interrupt Service Routine)
- ใช้ CHANGE interrupt บน Phase A ของ Quadrature Encoder ทั้ง 2 ตัว
- ได้ความละเอียด x2 (1012 counts/rev) จาก 506 PPR
- อ่านทิศทางการหมุนจาก Phase B
- ระยะทางต่อ 1 count = 0.000202 m (0.2 มม.)

#### 3.3.2 PID Velocity Controller (20Hz)
ทุก 50ms ESP32 จะ:
1. อ่าน encoder count แบบ atomic (ปิด interrupt ชั่วคราว)
2. คำนวณ velocity ปัจจุบัน (m/s) จาก encoder delta
3. คำนวณ PID output → แปลงเป็น PWM (0-250)
4. ใช้ dead-zone kickstart (MIN_START_PWM = 120) เพื่อเอาชนะ static friction
5. ส่ง Odometry data กลับ Pi5 ผ่าน Serial JSON

ค่า PID ที่ปรับแล้ว:
| Parameter | ค่า | หน้าที่ |
|-----------|-----|---------|
| Kp | 400 | ตอบสนองเร็วต่อ error |
| Ki | 50 | กำจัด steady-state error |
| Kd | 2 | ลด overshoot |
| MAX_PWM | 250 | จำกัดกำลัง |
| MIN_START_PWM | 120 | เอาชนะ motor dead-zone |

#### 3.3.3 Serial Communication Protocol (JSON, 115200 baud)

**คำสั่งรับจาก Pi5:**
```json
{"l": 0.10, "r": 0.10}
```
- `l` = left wheel velocity (m/s), `r` = right wheel velocity (m/s)

**ส่ง Odometry กลับ Pi5 (ทุก 50ms):**
```json
{"lt": 12048, "rt": 11932, "lv": 0.0991, "rv": 0.0988, "dt": 50}
```
- `lt`/`rt` = cumulative encoder ticks, `lv`/`rv` = velocity m/s, `dt` = loop time ms

**Safety Features:**
- Timeout 500ms — หยุดมอเตอร์อัตโนมัติถ้าไม่ได้รับคำสั่ง
- รองรับ runtime PID tuning — ไม่ต้อง flash firmware ใหม่
- Anti-windup ป้องกัน integral term สะสมมากเกินไป

### 3.4 พัฒนาซอฟต์แวร์ ROS2 — esp32_bridge Node

Bridge Node เขียนด้วย Python, ทำหน้าที่เป็นตัวกลางระหว่าง ROS2 กับ ESP32:

#### 3.4.1 cmd_vel → ESP32 (Inverse Kinematics)
1. รับ `/cmd_vel` (geometry_msgs/Twist) จาก teleop หรือ Nav2
2. แปลง linear.x + angular.z → v_left, v_right ด้วย diff-drive kinematics
3. ส่ง JSON ผ่าน Serial ไป ESP32

#### 3.4.2 ESP32 → Odometry (Dead-Reckoning)
1. อ่าน JSON จาก ESP32 ใน thread แยก
2. คำนวณ dead-reckoning (x, y, θ) จาก encoder delta
3. Publish `/odom` (nav_msgs/Odometry)
4. Broadcast TF: `odom` → `base_link`

### 3.5 SLAM — สร้างแผนที่ด้วย LiDAR

**📷 [แนะนำใส่รูป: ภาพแผนที่ที่สร้างจาก SLAM ใน RViz]**

ใช้ **slam_toolbox** (Online Asynchronous mode):
- รับ LiDAR scan จาก YDLiDAR X3 Pro (720 จุด/รอบ, 10Hz)
- ใช้ Scan Matching เพื่อจับคู่ scan กับแผนที่ที่กำลังสร้าง
- สร้าง Occupancy Grid Map (แผนที่ Grid ที่แสดงพื้นที่ว่าง/สิ่งกีดขวาง)
- รองรับ Loop Closure — แก้ไข drift เมื่อหุ่นกลับมาที่เดิม

**การแก้ปัญหา QoS Mismatch:**
YDLiDAR publish `/scan` แบบ BEST_EFFORT แต่ slam_toolbox subscribe แบบ RELIABLE ทำให้ไม่ได้รับ scan → แก้โดยพัฒนา `scan_qos_bridge` node เพื่อแปลง QoS อัตโนมัติ

### 3.6 Navigation — นำทางอัตโนมัติด้วย Nav2

**📷 [แนะนำใส่รูป: ภาพ Nav2 path planning ใน RViz — เส้นทางสีเขียว + costmap]**

| ส่วนประกอบ | Algorithm/Plugin | หน้าที่ |
|-----------|-----------------|---------|
| Path Planner | NavfnPlanner (Dijkstra) | วางแผนเส้นทาง global |
| Path Controller | RegulatedPurePursuit | วิ่งตามเส้นทาง smooth, ลด oscillation |
| Localization | slam_toolbox (Scan Matching) | ระบุตำแหน่งในแผนที่โดยใช้ LiDAR |
| Local Costmap | 3×3 m, rolling window | หลบสิ่งกีดขวางใกล้ |
| Global Costmap | Full map | วางแผนเส้นทางระยะไกล |
| Recovery | Spin, BackUp, Wait | กู้คืนเมื่อติดสิ่งกีดขวาง |
| Collision Monitor | FootprintApproach | ชะลอก่อนชน |

**พารามิเตอร์สำคัญที่ปรับ:**
- `desired_linear_vel: 1.0 m/s` — ความเร็วเต็มอัตรา
- `robot_radius: 0.10 m` — รัศมีหุ่นยนต์
- `inflation_radius: 0.15 m` — ระยะขยายสิ่งกีดขวาง (ลดลงเพื่อไม่บล็อกเส้นทาง)
- `transform_tolerance: 5.0 s` — เพิ่มสำหรับ Pi5 (แก้ TF timestamp drop)
- `allow_reversing: true` — ถอยหลังได้เมื่อจำเป็น

### 4.1 ทดสอบระบบ Motor Controller (ESP32)

**📷 [แนะนำใส่รูป: กราฟ PID response — target vs actual velocity]**

| รายการทดสอบ | ผลลัพธ์ |
|------------|---------|
| Encoder อ่านค่าถูกต้อง | ✅ 1012 counts/rev (x2 counting) |

## บทที่ 5: สรุปผลและข้อเสนอแนะ

### 5.1 สรุปผล

โปรเจคนี้ประสบความสำเร็จในการออกแบบและสร้างหุ่นยนต์เคลื่อนที่อัตโนมัติ (AMR) จากศูนย์ โดยบรรลุวัตถุประสงค์ทั้ง 4 ข้อ:

1. **Differential Drive** — สร้างหุ่นยนต์ที่เคลื่อนที่ได้อย่างแม่นยำ ด้วย PID Controller ที่ทำงานที่ 20Hz บน ESP32 ร่วมกับ Quadrature Encoder ที่มีความละเอียด 1012 counts/rev
2. **SLAM** — สามารถสร้างแผนที่ 2D ได้สำเร็จ โดยใช้ slam_toolbox ร่วมกับ YDLiDAR X3 Pro
3. **Autonomous Navigation** — สามารถวางแผนเส้นทางและนำทางอัตโนมัติได้ ด้วย Nav2 framework ที่ประกอบด้วย NavfnPlanner, RegulatedPurePursuit Controller และ Collision Monitor
4. **ROS2** — ประยุกต์ใช้ ROS2 Jazzy ได้อย่างครบถ้วน ทั้ง Topics, Services, Actions, TF, Launch System และ Lifecycle Manager

### 5.2 ปัญหาที่พบและวิธีแก้

| ปัญหา | สาเหตุ | วิธีแก้ |
|-------|--------|---------|
| มอเตอร์ไม่หมุนเมื่อ PWM ต่ำ | Motor dead-zone (static friction) | เพิ่ม MIN_START_PWM = 120 |
| SLAM ไม่สร้าง map | QoS mismatch (BEST_EFFORT vs RELIABLE) | พัฒนา scan_qos_bridge node |
| USB port สลับ (ESP32 ↔ LiDAR) | CP210x ไม่มี unique serial number | ใช้ udev rules lock ด้วย physical USB path |
| TF transform drop บน Pi5 | Computing latency | เพิ่ม transform_tolerance = 5.0s |
| Odometry drift เมื่อหมุน | Wheel slip + encoder resolution | ใช้ LiDAR-based localization (slam_toolbox) ชดเชย |

### 5.3 ข้อเสนอแนะสำหรับการพัฒนาต่อ

1. **เพิ่ม IMU** — ใช้ Extended Kalman Filter (EKF) เพื่อรวม odometry + IMU → ลด drift
2. **ปรับปรุง chassis** — ใช้วัสดุที่แข็งแรงกว่า เพิ่มเสถียรภาพ
3. **เพิ่ม Camera** — ใช้ depth camera สำหรับ 3D obstacle detection
4. **Multi-floor navigation** — พัฒนาระบบนำทางข้ามชั้น
5. **Web Interface** — พัฒนาหน้าเว็บสำหรับสั่งงานและ monitor หุ่นยนต์จากระยะไกล
